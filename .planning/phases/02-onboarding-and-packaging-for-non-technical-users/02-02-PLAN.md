<objective>
Replace the 10-second `Notice` flow that surfaces the device-flow user code with a persistent modal. Today the code disappears off-screen if the user blinks; the modal stays open until success, error, or cancel. Reused by both the welcome modal (plan 02-03) and the standalone Sign in command — one UI, two entry points.

Purpose: closes the second-highest install-friction gap. A semi-non-technical user can't be expected to read a 10-second toast at the corner of the screen, type a code into a browser, and get back to the plugin before the toast vanishes.
</objective>

    The existing `try/catch` around `await this.signIn()` in callbacks (e.g. `addCommand` for sign-in) keeps working: errors thrown from inside the modal propagate up via a rejected promise that the modal exposes.

    Pseudocode:

    ```typescript
    async signIn(): Promise<void> {
        if (this.startupBlocked) throw new Error(...);
        if (!this.auth) throw new Error(...);
        return new Promise((resolve, reject) => {
            new SignInModal(this.app, {
                startDeviceFlow: () => this.auth!.startDeviceFlow(),
                poll: (sessionId) => this.auth!.pollDeviceToken(sessionId),
                onSuccess: async (tokens) => {
                    this.data.tokens = tokens;
                    await this.saveAll();
                    new Notice("Huma: signed in.", 3000);
                    void this.runFullSync();
                    this.startPolling();
                    resolve();
                },
                onCancel: () => resolve(),
                onError: (err) => reject(err),
            }).open();
        });
    }
    ```
